## Anti-Patterns Verdict

**LLM Assessment:** 7/10 confidence — no new AI slop. The warm-paper + muted-rosewood palette remains the project's strongest counter-reflex. The 8 fixes actually reduced AI signals: glassmorphism is gone, focus rings are systematic, the collapsed plan is more intentional. The personalized content (real name, real textbooks, real chapters) continues to carry authenticity that generic output cannot.

**Detector Scan:** 4 hits, same 4 as before: `broken-image` (false — JS-populated lightbox src), `single-font` (false — 3 fonts loaded), `bounce-easing` (false — polaroidSpring is translate+opacity with snap easing), `layout-transition` (real but accepted trade-off on progress bar width).

**Browser Visualization:** Screenshot confirms no visual defects. Layout, typography, and interactive elements render correctly.

**Assessment B discovered our contrast fix isn't sufficient:**
- `--text-tertiary` #737373 on #F7F5F1 = **4.35:1** — misses WCAG AA by 0.15
- `--text-muted` #7A7A7A on #F7F5F1 = **3.94:1** — fails WCAG AA

These need another ~5% darkening. `#6E6E6E` would give 4.7:1. `#6B6B6B` would give 5.0:1.

## Overall Impression

The app is genuinely better. The 8 fixes removed rough edges that a daily user would have felt: focus rings show where the keyboard is, the dock feels like part of the page instead of floating above it, the plan doesn't scream at you on page load, and "reuse yesterday" eliminates re-entry for repeat study patterns.

But the critique score only moved +3 (24→27) because the **structural problems are structural.** The header information density, the checkin-below-progress ordering, the 60+ item dropdown, the hardcoded exam date — these are architectural decisions that can't be fixed with CSS tokens or focus rings. The surface is polished; the bones need work.

## What's Working

**1. "Reuse yesterday" with disabled state** (app.js lines 1146-1175). Correctly shows yesterday's duration + mood emoji when data exists, shows "6月24日 无记录" when it doesn't. The disabled state teaches the feature's existence while communicating its unavailability. Well-executed.

**2. Plan collapse toggle** (style.css lines 1595-1631). Clean button with chapter count, smooth arrow rotation on open/close, hover state consistent with the rest of the app. The plan is visually present but not imposed — this is the right compromise between "reference" and "action."

**3. Complete glassmorphism removal** (verified by Assessment B: 0 remaining `backdrop-filter`). All surfaces now use solid `var(--card)` or `var(--bg)`. The paper metaphor is now consistent from body texture through dock through photo wall container. This was a meaningful aesthetic coherence fix.

**4. Undo with IndexedDB cleanup** (app.js line 1266-1270). Rolling back a checkin also removes just-uploaded photos from IndexedDB. This is the kind of edge case handling that separates polished UIs from prototypes.

## Priority Issues

### [P0] Header information density — 200px of sticky chrome before any content

**Where:** `index.html` lines 21-114, `style.css` header section

**Why:** On a 375px phone at 16px base, the header consumes ~180-200px before the subject head appears. The four data-action buttons (export/import/photowall/settings) are used <1% of sessions but occupy permanent space. The overview strip shows 5 data points (date, checkin status, streak, pomodoro count, weekly hours) that could be a single line or a badge. The daily quote is decorative on every page load. All of this is sticky — it follows the user as they scroll, permanently consuming vertical space.

**Fix:** Three-step reduction: (1) Move data-actions into a "⋮" menu behind the avatar, (2) Collapse the overview strip behind the streak badge (tap streak → expand strip), (3) Move the daily quote into the subject page area (non-sticky, only visible on the active subject). Target: sticky header ≤100px on mobile.

**Suggested command:** `/impeccable distill index.html:header`

---

### [P0] Checkin form is below progress card in page order

**Where:** `index.html` lines 128-131 (and equivalent for all 4 subjects)

**Why:** The HTML order is: subject-head → progress → plan → checkin → today-log. The plan is now collapsed (improvement), but the progress card still sits between the user and their daily checkin form. The progress card shows one piece of information (% complete, X/Y chapters) that could be a compact stat in the subject head. It does not justify a full card of vertical space between arrival and action.

**Fix:** Move `math2-checkin` div before `math2-progress` div in HTML. The progress card can be a single row in the subject head: "📐 数学二 · 12% · 3/26章". This puts the daily action at the top of every subject page on first paint.

**Suggested command:** `/impeccable layout app.js:renderSubjectPage`

---

### [P1] `--text-tertiary` and `--text-muted` still fail WCAG AA

**Where:** `style.css` lines 15-16

**Why:** Assessment B calculated `#737373` on `#F7F5F1` = 4.35:1 (misses AA by 0.15). `#7A7A7A` = 3.94:1 (clearly fails). These tokens are used across 20+ selectors. Users with low vision or in bright ambient light will struggle with secondary text throughout the app.

**Fix:** `--text-tertiary: #6E6E6E` (4.7:1, passes AA with margin) and `--text-muted: #686868` (5.3:1, passes AA comfortably).

**Suggested command:** `/impeccable audit style.css:contrast`

---

### [P2] Exam date hardcoded with no configuration

**Where:** `app.js:renderCountdown`, `const target = new Date('2026-12-19')`

**Why:** Single point of temporal failure. If the exam is rescheduled or the app is used by another student, the countdown is wrong. The urgency styling (`urgency-warm` at 30 days, `urgency-near` at 7 days) is tied to this hardcoded date. After December 2026, the app permanently shows "考研已结束" with no post-exam mode.

**Fix:** Store exam date in `StorageManager` with a settings UI (date input in the settings modal). Default to `2026-12-19`. Allow the user to change it. Add an "exam passed" mode that switches to retrospective/archive view.

**Suggested command:** `/impeccable harden app.js:renderCountdown`

---

### [P2] Plan has no internal navigation — binary open/close

**Where:** `app.js:renderPlan`, `app.js:togglePlan`

**Why:** When expanded, the 819 plan shows 20+ phase cards with 60+ chapter rows. There's no table of contents, no sticky phase headers, no "scroll to current phase" shortcut. The toggle is all-or-nothing — the user can't expand just the phase they're currently working on.

**Fix:** Make each phase independently collapsible (phase header as toggle). Show current/active phase expanded by default, completed phases collapsed, future phases collapsed. Keep the master toggle for "expand all."

**Suggested command:** `/impeccable harden app.js:renderPlan`

## Questions to Consider

1. **What if the checkin form was a floating action button?** Instead of 4 pages × 4 forms, one unified checkin panel: pick subject → fill → submit. Zero page switching. The current architecture assumes subject-first navigation; actual behavior might be checkin-first.

2. **What if the breathing was opt-in?** A small "take a breath?" prompt after celebration instead of a forced 14-second dark overlay. Forced wellness is not wellness.

3. **What should this app become on December 20, 2026?** The countdown hits zero and the app has no post-exam mode. Is it an archive? A reflection tool? A habit tracker for the next goal?

4. **Is the plan useful as a wall of text?** 李嘉敏 already has this in her textbook. The app's value-add is progress tracking, not plan display. What if the plan showed only: current phase (expanded) + completed (collapsed) + future (hidden)?
