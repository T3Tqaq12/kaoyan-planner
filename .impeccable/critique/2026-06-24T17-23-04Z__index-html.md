## Anti-Patterns Verdict

**LLM Assessment:** 6/10 confidence — passes the AI slop test on category-reflex (warm paper + muted rosewood is the inverse of expected red/gold cram-school aesthetic), but shows second-order reflex leakage into the "editorial quiet luxury" aesthetic family. The warm paper background, serif headings, and muted accent are increasingly common in AI-generated "premium but not corporate" designs. However, the execution is genuinely personal — the user's real name, real school, real textbooks, real chapter data — and this specific application gives it authentic character.

**Detector Scan:** 4 hits, 3 false-positive: `broken-image` (false — lightbox src populated by JS), `single-font` (false — 3 fonts loaded: Caveat + Inter + Playfair Display), `bounce-easing` (false — polaroidSpring is translate+opacity with snap easing, not bounce), `layout-transition` (real but accepted — progress bar width transition, negligible perf impact on SPA).

**Cross-referenced findings the detector missed but manual/Browser scan caught:**
- `--text-tertiary` #949494 on #F7F5F1 = 2.7:1 contrast (fails WCAG AA)
- `--text-muted` #B8B8B8 on #F7F5F1 = 1.8:1 contrast (fails WCAG AA)
- Z-index magic numbers: 999 (celebration) and 1000 (breathing)
- Pervasive missing `:focus-visible` styles across all interactive elements except form inputs
- 9 of 13 interactive element types fail 44×44px touch target minimum
- Side-stripe border on `.phase-active` (style.css line 1538: `border-left: 2px solid var(--accent)`)
- Glassmorphism as default navigation style (dock uses `backdrop-filter: blur(20px)`)

**Browser Visualization:** Limited capture — Edge window confirmed open with correct page title, but PrintWindow API returned only minimal dock icon content. The 3 dock icons (📐💻📖📰) render correctly. Full-page visual inspection was attempted but unavailable due to window compositor limitations on this system.

## Overall Impression

This is a coherent, emotionally intelligent study companion that genuinely embodies its brand of "温暖 · 专注 · 陪伴" (warmth, focus, companionship). The warm-paper + muted-rosewood palette is the project's strongest design decision — it deliberately inverts the expected red/gold cram-school aesthetic. The personalized content (real name, real school, real textbooks) gives it authentic character that generic AI output cannot reproduce.

The core tension is between the design's ambition (editorial, quiet, warm, physical) and two sets of problems: **daily-use friction** that compounds over 200+ days (no "repeat yesterday," checkin form buried below plan, header steals mobile real estate), and **accessibility gaps** that make the app invisible to keyboard users and hard to read for anyone (missing focus styles, failing contrast on tertiary/muted text, small touch targets).

The app succeeds at being warm and personal. It partially succeeds at being focused. The header is the weak point of every page. The missing focus styles are the weakest point of the interaction design.

## What's Working

**1. The warm-paper palette and type system (style.css lines 7-68).** `#F7F5F1` background, `#7B5B5B` muted rosewood accent, Playfair Display serif + Inter sans + Caveat hand — a CSS custom property system where every token is used and relationships make sense. The editorial spacing scale (`--sp-2xs` through `--sp-2xl`) creates consistent rhythm. Paper noise at 3% opacity is exactly right.

**2. The undo toast pattern (app.js lines 1831-1861).** A 5-second window to undo a check-in, with clearly labeled button, properly cleaning up just-uploaded photos from IndexedDB (line 1267). This is more thoughtful error recovery than many production apps. The import confirmation dialog (lines 1863-1906) with merge/overwrite/cancel and day-count preview is equally well-executed.

**3. Genuinely personal, non-generic content.** The greeting names the actual user (李嘉敏, line 24), the school badge shows her university (西安工业大学, line 100), the plan data uses real textbooks and real chapter names with specific page counts and dates. This is "一人专属" (designed for one person) in practice — not abstracted, specific and warm.

## Priority Issues

### [P1] No "repeat yesterday" — daily friction compounding over 200+ days

**Where:** `app.js:renderCheckin` (lines 1025-1148), `app.js:handleCheckin` (lines 1171-1309)

**Why:** Each day, the user must re-enter duration, content, mood, and chapter for all four subjects from scratch. The form always defaults to duration=2, mood=easy, empty content, first-incomplete chapter. The user studies similar chapters on consecutive days with consistent duration patterns. Re-entering near-identical data across 4 subjects every day creates friction that accumulates to real resentment over months. The cognitive load assessment confirmed: "Working memory" failed — the user must remember yesterday's entries while filling today's form.

**Fix:** Store yesterday's entries per subject and offer a "复用昨日" (reuse yesterday) button above each check-in form. Pre-fill duration and mood from yesterday's values. Keep the smart chapter default (first incomplete) but make yesterday's chapter a one-tap option. Implementation: at form render time, read `dailyLogs[yesterday]`, filter by subject, populate form defaults.

**Suggested command:** `/impeccable harden app.js:renderCheckin`

### [P1] Check-in form buried below massive plan sections

**Where:** `app.js:renderSubjectPage` (lines 863-868), each subject page layout in `index.html` (lines 120-177)

**Why:** The page order is: subject-head → progress card → plan (all phases + all chapters + methods) → checkin form → today's log. For CS819, the plan alone spans 60+ chapters across 13+ phases. The check-in form — the most frequent daily action — is at the bottom of this long scroll. The information architecture puts reference content (the plan) above action content (the check-in form). Users open the app to check in, not to re-read the entire semester plan.

**Fix:** Reorder within each subject page: subject-head → checkin form → today's log → progress card → plan (collapsed by default with a "查看完整计划" expand button). This puts the daily action at the top, with reference content available but not imposed. Alternative: a fixed slide-up check-in panel accessible from the dock.

**Suggested command:** `/impeccable layout app.js:renderSubjectPage`

### [P1] Pervasive missing focus indicators — keyboard users completely locked out

**Where:** `style.css` — all `.btn`, `.btn-primary`, `.btn-ghost`, `.btn-glow`, `.dock-item`, `.tomato-icon-btn`, `.radio-icon-btn`, `.pw-filter`, `.pw-close`, `.modal-close`, `.mood-btn`, `.tomato-subject-btn`, `.hist-nav button`, `.checkin-photo-btn`, `.checkin-photo-remove`, `.breathing-skip`

**Why:** Only form inputs (`input`, `select`, `textarea`) have visible `:focus` styles. Every other interactive element — buttons, navigation, overlays, clickable cards — has zero focus indication. A keyboard-only user (or screen reader user) cannot see which element is focused when tabbing through the interface. This is the single largest accessibility gap. The dock — the primary navigation — has no `:focus-visible` style, making it impossible for keyboard users to switch subjects.

**Fix:** Add a consistent `:focus-visible` style to all interactive elements: `outline: 2px solid var(--accent); outline-offset: 2px;`. Create a shared focus-ring token and apply it systematically. For the dock, add visual feedback when a dock item receives focus. For close buttons and small controls, ensure the focus ring is visible against both light card backgrounds and dark overlays.

**Suggested command:** `/impeccable harden style.css:focus`

### [P1] Header information density on mobile

**Where:** `index.html` lines 19-114, `style.css` header section

**Why:** The sticky header contains: greeting + heart + sketch-eye, avatar with ring, pomodoro icon, radio icon, separator, school badge + streak badge + countdown badge, overview strip (date, status, streak, pomo count, week hours), daily quote, data-actions bar (4 buttons). On a 375px phone at 16px base, this consumes 180-220px before any study content appears. Many elements are informative but not actionable for the daily flow. The header competes with the primary task (daily check-in) for screen real estate.

**Fix:** Reduce sticky header to: greeting + date, streak badge, tomato + radio icon buttons. Move school badge to settings/about. Make overview strip collapsible (tap to expand). Move data-actions (export/import/photo wall/settings) to a slide-up sheet or settings modal. Keep daily quote in flow, not sticky. This reclaims 80-100px of vertical space on mobile.

**Suggested command:** `/impeccable distill index.html:header`

### [P2] Text-tertiary and text-muted fail WCAG AA contrast

**Where:** `style.css` design tokens `--text-tertiary: #949494` and `--text-muted: #B8B8B8` against `--bg: #F7F5F1`

**Why:** `#949494` on `#F7F5F1` = 2.7:1 (fails 4.5:1 AA minimum). `#B8B8B8` on `#F7F5F1` = 1.8:1 (fails even 3:1 large-text minimum). These tokens are used extensively: `.form-hint`, `.header-strip`, `.radio-dd-date`, `.log-entry-detail`, `.subject-head-meta`, `.tomato-ring-count`, and many more. Users with low vision or in bright ambient light (phone outdoors) will struggle to read secondary information throughout the app.

**Fix:** Darken `--text-tertiary` to at least `#737373` (4.6:1) and `--text-muted` to at least `#6B6B6B` (5.0:1). These are close to the current aesthetic while meeting AA. For decorative-only elements that truly don't need to be readable, use opacity on `--text` instead: `color: color-mix(in srgb, var(--text) 40%, transparent)`.

**Suggested command:** `/impeccable audit style.css:contrast`

---

### [P2] Glassmorphism dock + 5 overlay implementations → aesthetic coherence

**Where:** Dock: `style.css` lines 422-437; Overlays: `.celebration`, `.breathing-overlay`, `.tomato-overlay`, `.photowall-overlay`, `.modal-overlay`, `.radio-dropdown`

**Why:** The dock uses `backdrop-filter: blur(20px)` — a distinctly digital, iOS-7-era aesthetic that conflicts with the paper/scrapbook physical metaphor established by the body texture, card shadows, and polaroid frames. Separately, six overlay/panel types use different z-index values (140, 150, 500, 500, 999, 1000), different transition patterns, and different close mechanisms. Users must learn different dismissal patterns for each panel. The side-stripe border on `.phase-active` (`border-left: 2px solid var(--accent)`, line 1538) is a textbook absolute-ban violation.

**Fix:** (1) Replace glass dock with solid card background matching `--card`, using `--border` and `--shadow` for depth. Keep morphic bubble hover with solid `--accent-wash` background. (2) Replace `.phase-active` side-stripe with a full subtle border or background tint. (3) Create unified overlay system with consistent z-index scale (100/200/300/400/500), consistent close-button placement, consistent backdrop, and consistent ESC behavior.

**Suggested command:** `/impeccable polish style.css:dock+overlays`

## Cognitive Load Assessment

| Checklist Item | Status | Detail |
|---|---|---|
| Single focus | **FAIL** | Subject page shows progress + full plan + checkin + log simultaneously; header adds 11+ elements |
| Chunking | PASS | Form well-grouped; plan chunked by phase with clear headers |
| Grouping | **FAIL** | Header elements lack clear grouping; overview strip visually similar to badges but semantically different |
| Visual hierarchy | **FAIL** | Sticky header dominates viewport; plan section (reference) above checkin form (action); CTA `btn-glow` subtle on same card bg |
| One thing at a time | PASS | Subject pages isolate one subject; checkin → breathing → celebration is linear |
| Minimal choices | PASS | Dock: 4; Mood: 3; Photo wall filters: 5; Chapter dropdown uses optgroups to structure many options |
| Working memory | **FAIL** | User must remember yesterday's data while filling form; no reference visible; must scroll up to plan to verify chapter |
| Progressive disclosure | **MIXED** | Good: tomato config behind toggle, breathing gated to first daily, settings behind button. Bad: all phases + all chapters shown by default (60+ items for CS819), header shows everything at once |

**4 failures + 1 mixed = HIGH cognitive load.** The two largest contributors are the plan-above-checkin layout and the information-dense header. Both are structural fixes.

## Emotional Journey

**Opening (7:30 AM, phone):** Sees "李嘉敏 ♥" with sketch eye → feels seen. Daily quote loads → mildly motivating. Overview strip confirms today's date and streak. **Warm, mildly positive.** But must scroll past plan wall to find today's task.

**Filling check-in (4 PM, after studying):** Duration +/-, mood selection, chapter from long dropdown, optional content. **Neutral, task-focused.** Acceptable friction but no "reuse yesterday" means re-entering data.

**Submitting:** Button disables → "保存中…". Toast: "已打卡 ✅ [撤销]". If first checkin: screen goes dark for breathing overlay. **Interrupted.** Even with skip button, the sudden dark transition jars after warm paper world. If she skips → celebration: "🎉 打卡成功！". **Relieved, mildly celebrated.**

**Repeating for remaining subjects:** Same form, different subject. By fourth subject, novelty gone. **Routine, slightly tedious.**

**Browsing photo wall (weekend):** Polaroids scattered at random angles, springing in. **Nostalgic, pleasant.** Genuinely delightful.

**Arc:** Warm open → mild overwhelm (plan) → neutral task → jarring interruption (breathing) → pleasant relief → accumulating tedium → genuine delight (photo wall). The breathing overlay is the emotional valley — good intent, poor placement. Consider moving to standalone "take a breath" button rather than inserting between checkin and celebration.

## Minor Observations

1. **The dock label tooltip (hover-only, lines 501-524)** doesn't work on touch devices. The `aria-label` attributes partially compensate but aren't visible.
2. **`#breathingSkip` color** (`rgba(255,255,255,0.3)`, line 1831) fails contrast against the dark background at ~2.1:1 — crosses from "subtle" to "invisible."
3. **Celebration auto-dismiss (2000ms, line 1807)** is too fast to read "坚持就是胜利，明天继续加油 ✨" — consider 3000-3500ms or dismiss-on-tap.
4. **`.header-avatar` HTML attributes** (width=106 height=160, line 39) conflict with CSS (width:42px) — misleading.
5. **9 of 13 interactive element types fail 44×44px touch target.** Close buttons (28px, 20px), filter chips (12×24px), and small action buttons are the worst offenders.
6. **Custom scrollbar at 4px width (line 2281)** is nearly impossible to grab on high-DPI touch screens.
7. **The hardcoded exam date** means the countdown will permanently show "考研已结束" after December 2026.
8. **Z-index 999/1000** are magic numbers — replace with semantic scale.
9. **`.pw-filters` hidden scrollbar** (`scrollbar-width: none`) means users may not discover horizontally-scrollable filter chips.

## Questions to Consider

1. **What if the user studies two chapters in one sitting?** The check-in form allows only one chapter per submission. Is this the intended behavior, or should multi-chapter checkin be supported?

2. **Should the breathing exercise live between checkin and celebration?** Its intent (mindfulness during high-pressure prep) is aligned with the brand, but it creates an emotional speed bump at the moment of accomplishment. Consider: a standalone "take a breath" button, or offering it at app-open rather than post-checkin.

3. **Is the plan reference or action?** If reference: collapsible and below checkin. If action (marking chapters complete): the checkin form's chapter dropdown is redundant with the plan's chapter list. Pick one interaction model.

4. **What happens after December 2026?** The hardcoded exam date, the "考研已结束" badge, and no archive mode suggest the app has no post-exam life. Is this intentional, or should it transition to a retrospective/archive mode?

5. **How critical is keyboard accessibility?** PRODUCT.md says "手机为主（拍照打卡），桌面为辅（查看规划）" — mobile-first, desktop secondary. But if the user ever reviews plans or browses history on desktop, keyboard navigation matters. The current state (zero focus styles on buttons/nav) makes desktop use effectively impossible for keyboard users.
